진행 과정

1. 작업 세분화

  • # 메인페이지
  • # 서브페이지1 ( 사업영역 - 크리에이티브 ) + 헤더 + 푸터
  • # 서브페이지2 ( 플레이디 - ABOUT )

업무 담당은 업무량을 고려해서 구현 할 움직임이 많은 ‘메인페이지’ 하나, 내용이 적은 ‘서브페이지1’에서 ‘헤더+푸터’까지 하나, 컨텐츠가 많은 ‘서브페이지2’ 하나로 팀원 총 3명이 위 내용처럼 세 구역으로 업무를 분담 했습니다.

2. 코딩컨벤션 작성 및 공유

작업을 시작하기 전에 폴더명, 폴더위치, 선택자 작성법 등을 맞춰 경로나 merge 후의 혼란을 최소화 하고자 했습니다.

3. header+footer 공유 후 각 페이지에 적용

각 페이지에서 동일하게 적용되는 header+footer를 먼저 공유해 각 페이지에 적용했지만 페이지마다 header+footer가 약간씩 다르다는 것을 알고 각 페이지에 맞게 수정된 것을 각 페이지 담당자에게 따로 공유했습니다.

발생한 문제를 해결하기 위한 소통방식에 대해 팀원들과 함께 고민해 볼 수 있었고 문제 발생시 서로에게 문제를 공유해서 각자 찾아본 방안을 제시해 문제를 함께 고민/해결 하고자 했습니다.

4. 각 브랜치에 push후 main브랜치에 merge

작업 완료된 페이지를 각 브랜치에 push하고 main브랜치에 merge해서 각 페이지를 하나로 합쳐 전체적인 작동을 확인했고 header에 문제가 발생한 것을 확인 했습니다.

header에 마우스를 올렸을때 부드럽게 내려와야 할 서브메뉴의 백그라운드 영역이 작동하지 않았는데 자바스크립트로 작업했던 부분을 css에서 hover로 수정해 문제를 해결했습니다.

문제 & 해결

  • header + footer

    스크롤시 헤더와 서브메뉴 높이의 변화
    자바스크립트에서 스크롤 이벤트를 작업해 주었으며 아래로 스크롤시 classList.add로 클래스를 추가해 header의 높이가 줄어들고, 화면 위 끝까지 닿으면 classList.remove로 클래스를 제거해 header의 높이가 처음상태로 돌아가도록 했습니다.

    서브메뉴 영역
    '서브메뉴1'의 두번째 목록 선택시 '서브메뉴2'의 목록이 선택되는 문제가 발생해서 ‘서브메뉴1’에 mouseover 되어있는 동안은 다른 서브메뉴가 선택되지 않도록 overflow: hidden이 적용되고 mouseout 될 경우 overflow: visible이 적용되도록 했습니다.

    위로가기 버튼이 나타나는 위치
    화면 오른쪽 하단에 고정되어 있는 위로가기 버튼이 안보였다가 아래로 스크롤 할때 나타날 수 있도록 자바스크립트에서 offsetTop을 사용해 해당 위치에서 보여지도록 했습니다.

    메뉴 버튼
    메뉴 버튼 클릭시 자바스크립트에서 toggle로 '.active' 클래스명을 추가/제거 되는 방법으로 서브메뉴가 작동하도록 했고, '.active'에 나머지 영역은 'display:none'으로 설정해 서브메뉴가 열리면 세로 스크롤이 생기지 않고 화면 전체를 채울 수 있도록 했습니다.

  • 서브페이지1 (사업영역 - 크리에이티브)

    이미지 컨텐츠 마우스 이벤트
    이미지 위에 마우스를 올리면 텍스트가 나타나고 영역이 줄어들지만 이미지의 사이즈는 커질 수 있도록 css에서 hover와 opacity를 사용했습니다.

Review

  • Scroll Event

    스크롤에 따라 헤더에 움직임 변화가 많아서 스크롤 상황에 맞게 헤더를 변화 시켜야 했고 'transition' 값을 줬는데도 기대한 움직임이 작동하지 않아 많이 헤맸지만 JavaScript에서 addEventListener의 scroll 이벤트와 scrollTop으로 스크롤 상황에 맞게 헤더가 변할 수 있게 작업했습니다.

    'display: none'으로 작업해서 'transition' 이 작동하지 않았고 'height' 값으로 높이를 조절해 'transition' 이 작동할 수 있도록 했습니다.

  • Sub Menu

    이전부터 서브 메뉴 만드는게 어려워서 이번에 담당을 지원했고 담당한 부분 중 가장 많은 시간을 들이기도 했는데

    풀지 못한 문제에 대해 놓지 않고 해결 방안을 찾아보는 과정을 경험해 볼 수 있었습니다.

  • GitHub의 organization을 이용한 협업

    팀프로젝트용으로 새로운 organization을 생성 후 팀원을 초대해서 저장소를 함께 사용했고, 각자 담당한 영역별로 새로운 브랜치를 생성해서 작업물을 업로드하면 최종 작업물을 merge 해 완성했습니다.

    팀프로젝트인 만큼 소통에 대해 많은 고민을 해볼 수 있었고 하나의 결과물을 만드는 팀으로써 다른 사람이 담당한 업무에도 내 담당 업무만큼 관심을 갖고 문제를 함께 고민해주니 정체되는 구간 없이 진행이 원활할 수 있었습니다.
